Reporter Trần Hoàng Quân Incident Overriding image tag caused mixed pod versions (old pod uses old image, new pod uses new image with same tag) Priority HIGH Affected services Smart TV front-end Executive summary This is a front-end application where every git tag triggers CI/CD to rebuild and produce a new Docker image. After we moved the git repository to a different host, tags were recreated (re-tagged). That re-tagging triggered CI/CD again, which rebuilt the front-end and pushed a new image using the same existing image tag . Because the Kubernetes deployment still referenced that tag, some pods continued running the old image while newly created pods used the new image content under the same tag. Combined with CDN caching, this sometimes caused the CDN to cache wrong or mismatched front-end assets (HTML/JS/CSS). As a result, users could see inconsistent or broken UI depending on which pod and which cached assets they hit. We mitigated by building and pushing a new image with a unique, immutable tag , updating the deployment to use that tag so all pods ran the same version, and ensuring the CDN served the correct assets. Incident report Leadup New build reused an existing Docker image tag. Deployment applied; some pods restarted and pulled the new image, while others continued running the old image with the same tag. (screenshot/image to be added later) Fault Reusing a mutable tag allowed different image digests under one tag. Kubernetes did not force all pods to pull the changed tag, resulting in mixed versions. Detection Operation team reported that many smart TV cannot display Response DevOps team created new tag to trigger CI/CD build new image Recovery Timeline (UTC) Five whys Why mis-behaving pods? Mixed image versions within one deployment. Why mixed versions? New build reused an existing image tag. Why tag reused? No enforced immutable tagging policy in production. Why no policy/tooling? Tagging standard not automated in CI/CD or registry policies. Why not addressed earlier? Previous non-prod issues weren’t converted into preventive actions. Blameless root cause Process and tooling allowed mutable image tags in production, enabling different image digests to run simultaneously in the same deployment. Backlog check Add items to define immutable tagging, enforce unique tags in CI/CD, and verify single image digest per deployment. Related incidents Earlier inconsistencies in lower environments after tag reuse; not previously formalized as incidents. Lessons learned Use immutable image tags in production (e.g., version+build or git SHA). Block deployments that reference mutable tags.
