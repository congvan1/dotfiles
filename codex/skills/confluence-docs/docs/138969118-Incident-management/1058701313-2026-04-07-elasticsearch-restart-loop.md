📋 Tóm tắt (Summary) Reporter Inform Incident Elasticsearch container rơi vào trạng thái restart loop do full root disk trên vm-vtvcab-prod-es-data-002 Priority HIGH Affected services Elasticsearch cluster trên vm-vtvcab-prod-es-data-002 - Các ứng dụng sử dụng node này để search/indexing Executive summary Ngày 2026-04-07, Elasticsearch container es-vm-vtvcab-prod-ES-data-002 trên host vm-vtvcab-prod-es-data-002 rơi vào trạng thái restart loop. Filesystem root /dev/sda1 đạt 100% dung lượng sử dụng, chủ yếu do dữ liệu lớn nằm trong Docker overlay2 storage của Elasticsearch container. Root disk bị full khiến Elasticsearch container lỗi và liên tục restart. Sự cố được khắc phục bằng cách dọn dẹp các Docker image không còn sử dụng (dangling images), dọn dẹp các container không còn sử dụng, xóa thư mục dữ liệu tạm (temporary data) quá lớn trong overlay2 , tạo lại thư mục với quyền phù hợp, sau đó start lại container. Dịch vụ được khôi phục sau khi giải phóng đủ dung lượng đĩa. 📝 Incident report Instructions Report ⚠️ Leadup Elasticsearch node vm-vtvcab-prod-es-data-002 đang chạy nhiều Elasticsearch-related images và containers, bao gồm: - quantran/elasticsearch-vietnamese:8.7.0 (container: es-vm-vtvcab-prod-ES-data-002 ) - elasticsearch-custom:8.7.0 (container: es-vm-vtvcab-prod-ES-data-002_backup ) - Theo thời gian, dữ liệu và temporary files tích lũy trong Docker overlay2 storage. Có alert: 🙅‍♀️ Fault Phân vùng root /dev/sda1 bị full hoàn toàn (100% usage). - Docker không thể ghi dữ liệu container / logs / temporary files cho Elasticsearch container 053a4464b290 một cách bình thường. - Điều này làm container bị lỗi và liên tục restart (trạng thái Restarting (137) ). 🥏 Impact Elasticsearch container es-vm-vtvcab-prod-ES-data-002 không ổn định và liên tục restart. - Khả năng làm giảm capacity và tính sẵn sàng (resiliency) của Elasticsearch cluster (một data node không ổn định). - Ảnh hưởng tới người dùng cuối (tùy mức độ sử dụng ứng dụng): - Giảm performance hoặc availability của search. - Có thể gây trễ hoặc lỗi indexing đối với các service dùng node này. 👁 Detection Phát hiện sự cố bằng cách kiểm tra host và trạng thái container: - Lệnh df -h cho thấy /dev/sda1 ở mức 100%: - /dev/sda1 94G 90G 0 100% / - docker system df -v cho thấy Elasticsearch containers sử dụng dung lượng lớn: - Container 053a4464b290 dùng khoảng 49.5GB, trạng thái Restarting (1) . - Phân tích chi tiết dung lượng đĩa: - du -h --max-depth=1 /var/lib/docker cho thấy ~120GB được sử dụng, trong đó overlay2 chiếm 119GB. - du -h --max-depth=1 /var/lib/docker/overlay2/.../diff cho thấy ~47GB trong thư mục /tmp bên trong container layer. - Cần cải thiện: bổ sung monitoring & alerts cho disk usage trên các Elasticsearch node quan trọng (ví dụ alert ở ngưỡng 80%, 90%). 🙋‍♂️ Response Xác minh tình trạng đĩa: - Chạy df -h để kiểm tra và xác nhận /dev/sda1 bị full. - Điều tra sử dụng Docker: - sudo docker system df -v để xác định image/container chiếm nhiều dung lượng. - Xác định thư mục Docker chiếm root disk: - sudo du -h --max-depth=1 /var/lib/docker (tổng 120G, overlay2 119G). - Xác định cụ thể overlay2 layer có ~47G trong diff/tmp . - Thực hiện các bước cleanup: 1. Xóa Docker images không còn dùng (dangling images): - sudo docker image prune -a 2. Dừng Elasticsearch container gặp sự cố: - sudo docker stop 053a4464b290 3. Xóa thư mục temp quá lớn của overlay layer tương ứng: - sudo rm -rf /var/lib/docker/overlay2/968f9af28f3.../diff/tmp 4. Tạo lại thư mục /tmp với quyền đúng: - sudo mkdir /var/lib/docker/overlay2/968f9af28f3.../diff/tmp - sudo chmod 1777 /var/lib/docker/overlay2/968f9af28f3.../diff/tmp 5. Start lại container: - sudo docker start 053a4464b290 - Sau các bước trên, container start thành công và disk usage trở lại mức an toàn. 🙆‍♀️ Recovery Sau khi xóa temporary data và dọn dangling images, dung lượng trống trên /dev/sda1 đủ để hệ thống hoạt động bình thường. - Start lại Elasticsearch container thành công, node join lại vào cluster. - Ảnh hưởng tới người dùng được giảm thiểu khi node ổn định trở lại (thời gian phục hồi chính xác: TBD , cần bổ sung). - Cần thêm: xây dựng cơ chế rotation/cleanup tự động cho log và temporary data của Elasticsearch containers để tránh full root disk. ⏲ Timeline 🔎 Five whys root cause identification 🌱 Blameless root cause Nguyên nhân gốc rễ của incident là việc quản lý capacity disk trên Elasticsearch data node vm-vtvcab-prod-es-data-002 chưa đầy đủ. Docker overlay2 storage của Elasticsearch container tích lũy lượng lớn temporary data trong /tmp , dẫn đến root filesystem /dev/sda1 bị full và container liên tục restart. Không có cơ chế alert chủ động hoặc quy trình cleanup tự động cho loại dữ liệu này, khiến dung lượng đĩa tăng dần tới 100% trước khi được phát hiện và xử lý. ✅ Backlog check Kiểm tra backlog hiện có xem đã có task nào liên quan tới: - Thiết lập disk usage monitoring/alerts cho các Elasticsearch nodes. - Triển khai log & temp file rotation cho containers. - Tách dữ liệu/overlay storage ra khỏi root partition. - Nếu đã có các task tương tự, cần nâng mức độ ưu tiên và đưa vào kế hoạch xử lý sớm. 🗃 Related incidents Kiểm tra các incident trước đây liên quan: - Full disk trên Elasticsearch hosts hoặc các Docker hosts khác. - Elasticsearch containers lỗi do vấn đề disk hoặc temp folder. - Nếu có các incident tương tự, so sánh giải pháp đã áp dụng và lý do pattern này vẫn tái diễn (ví dụ: alert chưa được implement, không tăng capacity, chưa có cleanup tự động). 🤔 Lessons learned Các critical data node có thể bị gián đoạn do vấn đề disk ngay cả khi CPU/RAM vẫn bình thường. - Docker overlay2 có thể âm thầm tích lũy lượng dữ liệu rất lớn; cần kiểm tra và cảnh báo thường xuyên. - Thư mục temp/log của Elasticsearch bên trong containers cần được quản lý chặt chẽ (rotation, retention, cleanup). - Việc tách Docker data khỏi root filesystem giúp giảm rủi ro node bị sập hoàn toàn khi một thành phần đơn lẻ làm full disk. ✍ Follow-up tasks Cấu hình monitoring và alerts cho disk usage trên vm-vtvcab-prod-es-data-002 và các Elasticsearch nodes khác (ví dụ: threshold 70/80/90%). - Triển khai cơ chế log & temp file rotation tự động cho Elasticsearch containers để kiểm soát dung lượng /tmp và logs. - Đánh giá phương án chuyển Docker data ( /var/lib/docker ) sang partition hoặc disk riêng với dung lượng phù hợp. - Viết và chuẩn hóa SOP (Standard Operating Procedure) xử lý sự cố full disk trên Elasticsearch nodes. - Thiết lập lịch cleanup định kỳ cho Docker images/containers không dùng (cron job hoặc trong quy trình CI/CD housekeeping). - Tạo các Jira issues liên quan và link vào đây (ví dụ: SD-XXXX , OPS-YYYY ).
