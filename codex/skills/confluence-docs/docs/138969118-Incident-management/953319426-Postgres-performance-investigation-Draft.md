Status In Progress Blue Creator Contributors Table of contents Approvers Context This doc consolidate the investigation of Postgress performance in the incident: 2026-01-07 High response time in GraphQL components Investigation During live event, we observed the very high response time in GraphQL components of the Subscription service. When analyze the trace I saw that: The trace shows pool.acquire taking ~5 seconds. This pool refers to the Client-Side Connection Pool inside the Go application (the pgxpool ). What is happening: The Go application is configured with a maximum connection limit (e.g., pool_max_conns = 10 ). The "Session Mode" Penalty: In Session Mode, every active client connection requires a real backend connection. When N+1 query pattern tries to spin up many concurrent queries, but the Go app is limited to 10 connections, some of them must wait inside the Go runtime until a connection is returned. Here is the image show the trace of slow request First, the apollo router query users in identity-service-graphql-server. Then, with each user, it fetches every subscription of a user from the Subscription subgraph In the Subscription subgraph, the reference resolver for Subscription doesn't take a list of users but rather a single user. That would result in N database calls And here the number of go routing spawned during peak traffic Sample to simulate So I create a small test to compare between transaction mode and session mode so simulate that at https://github.com/QuanTran91/postgres-transaction-mode-vs-session-mode This benchmark simulates a real-world scenario: multiple Go servers all trying to talk to the same PostgreSQL database through PgBouncer. Each server has its own connection pool, and we're intentionally creating a bottleneck to see what breaks first. Affected component Go server The go server use golang-migrate to migrate the database schema, which uses advisory locks which not supported in transaction mode. So we have to separate the component which involve only in db migration (presync component) https://github.com/golang-migrate/migrate/issues/1001 SpiceDB SpiceDB supports pgbouncer in transaction mode: https://github.com/authzed/spicedb/pull/1994 https://authzed.com/docs/spicedb/concepts/datastores However, db migration does not support. https://github.com/authzed/spicedb-operator/issues/338 So the solution is enable skipMigrations flag in cluster, then create another k8s job (sync hook) to run migration. DevOps will feed migration uri for that job. Keycloak Keycloak does not support PgBouncer transaction mode Keycloak's JDBC driver defaults to using server-side prepared statements for performance. This can be fixed in Pgbouncer via update max_prepared_statements > 0 Keycloak uses Liquibase to manage its database schema. Liquibase often uses database locks which does not support in transaction mode. Migh be we can create k8s job to migrate database same as SpiceDB Flink Flink's JDBC driver relies heavily on Server-Side Prepared Statements for performance. For read side (source): Set max_prepared_statements > 0 on Pgbouncer can fix it. For the write side (sink): XA Transactions (to archive Exactly-Once semantics) are not supported by PgBouncer. https://nightlies.apache.org/flink/flink-docs-release-1.18/docs/connectors/datastream/jdbc/ At the moment, only Notification service use Flink Table API (via Postgres JDBC connector) Gorse Similar to Go Server Caveat: Need more confirmation
