# Seenow service deployment runbook

This runbook records the conventions found in:

```text
/Users/van/workspace/work/clients/Cellutions/devops/seenow-argocd/apps-vng
```

It is intended for setting up or deploying a new Seenow service, using
`homepage-service`, `vod-service`, and `channel-service` as reference
implementations.

## Mental model

The repository is a Kustomize application catalog. A service is assembled from
Helm charts, environment-specific values, ConfigMaps, SealedSecrets, and
shared platform resources.

```text
service/base/                 reusable component values and base config
service/overlays/dev/         dev secrets, env files, image tags, overrides
service/overlays/stg/         staging equivalents
service/overlays/prd/         production equivalents
00-shared-config/             shared env, service accounts, common secrets
04-pulsar-resources/          Pulsar tenants, namespaces, topics
05-hooks/                     ArgoCD sync/presync hook charts and credentials
06-gorse/                     shared Gorse chart values
08-es-postgres-pulsar-connector/ shared connector values
13-kustomize-component/       reusable post-render patches/components
01-apisix-route/              external/API gateway routes
../base/charts/               shared Helm charts
```

The normal deployment flow is:

```text
build/publish service images
        -> keep overlay image tags at latest; CI/CD replaces them
        -> render and validate Kustomize
        -> sync the ArgoCD Application
        -> verify rollout, jobs, routes, and dependencies
```

## Service layout

Use the existing service names and component names as the naming contract.

| Service | Main components in dev | Supporting components |
|---|---|---|
| homepage | GraphQL server, command ingress, upload server | ES/Postgres/Pulsar connector, sync hook, presync hook, trending-keywords CronJob, block-precompute CronJob, Gorse |
| vod | GraphQL server, command ingress | Cassandra projector, sync hook, presync hook, Gorse |
| channel | GraphQL server, command ingress, upload server | EPG import CronJob, sync hook, presync hook, ES connector, ECST connector |

The exact component list is environment-specific. For example, staging and
production can enable projectors that are omitted or commented out in dev.
Always copy the nearest service overlay and compare all three environments
before adding a component.

## How to create a new service

### 1. Create the service directories

```text
<service>/
  base/
    kustomization.yaml
    <component>-values.yaml
    <service>-env.properties
  overlays/
    dev/
      kustomization.yaml
      dev-<service>-env.properties
      <component>-values.yaml
      <secret>-sealed-secret.yaml
    stg/
    prd/
```

`homepage-service/base/kustomization.yaml` and `vod-service/base/kustomization.yaml`
contain no direct resources because their workloads are generated by Helm.
`channel-service/base/kustomization.yaml` additionally includes shared base
Kustomize resources. Choose the pattern that matches the service runtime.

### 2. Select the chart per workload

The shared charts are referenced with `helmCharts` and this chart root:

```yaml
helmGlobals:
  chartHome: ../../../../base/charts
```

Common chart choices:

| Workload | Chart |
|---|---|
| HTTP/GraphQL/command/upload service | `standard-service` |
| One-shot sync or presync task | `job` |
| Scheduled import or maintenance task | `cronjob` |
| Recommendation service | `gorse` |
| Flink application where required | `flink-cluster` or the service's existing Flink pattern |

Use the service's base values for stable configuration and the overlay values
for environment-specific endpoints, annotations, resources, and runtime
switches. Keep every new service image tag in `*-values.yaml` as `latest`;
the CI/CD process replaces it with the correct immutable tag.

### 3. Define stable release names

Each component gets a predictable `releaseName` and usually an explicit
`fullnameOverride`, for example:

```yaml
- name: standard-service
  releaseName: <service>-graphql-server
  valuesFile: ../../base/graphql-server-values.yaml
  additionalValuesFiles:
    - graphql-server-values.yaml
```

Keep these names consistent across:

- Helm `releaseName` and `fullnameOverride`
- `nameService` and `nameComponent`
- image repository paths
- `envFrom` Secret and ConfigMap references
- service-to-service DNS names
- labels used by performance or monitoring components

### 4. Reuse shared configuration and secrets

Use `configMapGenerator` for non-secret environment configuration. Existing
services generally use stable ConfigMap names:

```yaml
configMapGenerator:
  - name: <service>-cm
    options:
      disableNameSuffixHash: true
    envs:
      - dev-<service>-env.properties
      - ../../base/<service>-env.properties
      - ../../../00-shared-config/overlays/dev/dev-pulsar-config-env.properties
      - ../../../00-shared-config/overlays/dev/dev-s3-env.properties
```

Reuse the shared environment ConfigMaps rather than creating duplicate
Pulsar/Kafka/Schema Registry configuration. Use the matching overlay for each
environment:

```text
00-shared-config/overlays/dev/
00-shared-config/overlays/stg/
00-shared-config/overlays/prd/
```

Reuse the existing Redis service Secret pattern as well. Copy the nearest
service Secret manifest, rename the resource and filename to the new service,
then update only the references that are service-specific. Add image-proxy,
S3, or connector env files only when the component needs them. Keep Gorse
configuration in a separate ConfigMap when using the Gorse chart.

Before creating a new secret, search the service directory and shared config
for an existing one. If it does not exist, create a file using the existing
environment/service naming convention, for example:

```text
dev-<service>-<purpose>-sealed-secret.yaml
```

For a genuinely new secret, the initial manifest may use Kubernetes `Secret`
with `stringData` while being created locally or through the approved secret
workflow. Seal it before committing. The committed manifest must be a
`SealedSecret`; never commit real passwords, access keys, private keys, or
tokens in plaintext.

### 5. Add service dependencies

Declare the runtime dependencies explicitly in the component values:

- `envFrom` references to service SealedSecrets and generated ConfigMaps
- internal Kubernetes DNS names for other Seenow services
- Pulsar and Kafka/Schema Registry settings from the environment shared config
- S3 settings from shared config plus service-specific bucket names
- Cassandra/Postgres/Redis settings from the service overlay
- Gorse settings only for services that use recommendations

Use placeholders and the target environment's existing naming convention; do
not copy an address from another environment into a new overlay.

### 6. Add event-processing components only when needed

For dev, do not deploy a standalone Flink application when the environment
uses the shared Flink session cluster. Enable only the service components that
run outside that session, such as command-ingress, GraphQL, upload, jobs, or
connectors.

For stg/prd, projectors that run as Flink applications need:

- service account and workload node selection
- checkpoint and savepoint locations per environment
- initial savepoints, when a migration requires one
- contract/topic annotations and revision labels
- PodMonitor/profile scraping settings
- connector-specific database and Pulsar configuration

Do not enable a projector or connector in an overlay merely because a base
values file exists. The overlay `kustomization.yaml` is the enablement list.

### 7. Add the overlay

The overlay usually contains:

1. SealedSecret resources.
2. `helmCharts` entries for enabled components.
3. ConfigMap generators.
4. Shared Kustomize components.

Most service overlays use:

```yaml
components:
  - ../../../13-kustomize-component/change-port
```

That component standardizes the generated service health and metrics ports,
probes, and PodMonitor behavior. Reuse it unless the workload has a documented
exception.

## Validation before sync

Run rendering from the repository root. Helm-backed Kustomizations need Helm
enabled:

```bash
kubectl kustomize --enable-helm apps-vng/<service>/overlays/dev \
  > /tmp/<service>-dev-rendered.yaml
```

Then inspect the rendered output for:

- expected Deployments, Services, Jobs, CronJobs, and PodMonitors
- expected namespaces and stable names
- image repositories and `latest` tags in the new `*-values.yaml` files
- `envFrom` references that exist in the same overlay or shared config
- no accidental plaintext secret values
- no disabled/commented component that was unintentionally re-enabled

When a cluster context is available, validate server-side and inspect the
resulting resources:

```bash
kubectl apply --dry-run=server -f /tmp/<service>-dev-rendered.yaml
kubectl -n sn-dev-workload get deploy,job,cronjob,svc,podmonitor
kubectl -n sn-dev-workload get pods
```

Deploy by syncing the corresponding ArgoCD Application. Use direct
`kubectl apply` only for an explicitly approved emergency or debugging action;
otherwise ArgoCD self-heal will reconcile the declared source.

## Quick reference: the three reference services

### homepage-service

The dev overlay enables GraphQL, command ingress, upload, ES/Postgres/Pulsar
connector, sync and presync hooks, two CronJobs, and Gorse. It consumes shared
Pulsar, S3, and image-proxy configuration and references internal VOD and
channel GraphQL services.

### vod-service

The dev overlay enables GraphQL, command ingress, Cassandra projector, sync
and presync hooks, and Gorse. Staging/production additionally show Redis and
Gorse projector patterns, so those should be added deliberately per
environment rather than assumed from dev.

### channel-service

The dev overlay enables GraphQL, command ingress, upload, EPG import CronJob,
sync and presync hooks, and ES/ECST Postgres/Pulsar connectors. It also has
extra DRM, EPG, and Elasticsearch configuration. Its base Kustomization includes
shared base resources, unlike the simpler homepage and VOD bases.

## Common failure modes

| Symptom | Likely cause |
|---|---|
| ArgoCD has no Application | Parent app-of-apps registration is missing |
| Helm render cannot find chart | `helmGlobals.chartHome` or relative values path is wrong |
| Pod cannot start | Secret/ConfigMap name does not match `envFrom` |
| Service is healthy but unreachable externally | APISIX route was not added or generated |
| Connector/projector starts unexpectedly | Component was added to overlay `helmCharts` unintentionally |
| Config changes are not observed | Stable ConfigMap name, rollout/reloader behavior, or sync is missing |
| Hook blocks deployment | Hook values, sync wave, ServiceAccount, or ArgoCD credentials are wrong |
| Pulsar processing stalls | Check application logs and topic/subscription metrics before changing broker settings |
